# HSAH 信息化打磨季度方案设计

Date: 2026-06-29
Scope: Hyperscaler Asset Hub
Status: approved design

## 1. 背景与目标

Hyperscaler Asset Hub 当前已经具备内容资产平台的主体骨架：前台公共门户、资产列表与详情页、后台管理壳、资产 CRUD、发布状态、模板、RBAC、访问策略、策略模拟、全站中英文国际化，以及基础测试覆盖。

本次信息化打磨的目标，是在一个季度内把项目从“资产展示 Demo”升级为“可信的内容资产运营平台”。重点不是堆叠大量外围功能，而是补齐运营闭环与用户体验闭环，使平台看起来、用起来、管起来都更像真实可运营系统。

成功标准以“平台可信度”为核心：

1. 管理员知道哪些资产需要处理、为什么不能发布、发布后表现如何。
2. 用户能快速发现资产、评估价值，并完成收藏、反馈或访问申请。
3. 平台能用质量分、审核记录、审计日志和数据看板证明内容可靠、流程可控、运营可复盘。

## 2. 设计方向

采用“运营闭环优先 + 前台可信体验同步增强”的双端闭环方案。

### 2.1 为什么不只做后台

只强化后台可以最快提升管理能力，但前台仍会停留在内容浏览层，用户无法形成明确的评估和转化动作，平台可信度提升不完整。

### 2.2 为什么不先做智能化

智能质检、智能推荐和个性化首页适合作为后续增强，但当前基础流程尚未完全闭环。过早建设智能化能力会让方案显得悬空，也会增加实现复杂度。

### 2.3 推荐方案

后台形成资产生产、审核、发布、访问申请、数据复盘闭环；前台形成发现、评估、收藏、申请、反馈闭环。两端共同支撑平台可信度。

## 3. 季度能力地图

### M1：运营闭环打底

目标：让后台从“可管理”变成“可运营”。

交付内容：

1. 升级后台首页为运营工作台，展示待审核资产、待补全资产、近期发布、异常内容、快捷操作和平台健康度。
2. 补资产审核流，在现有 `draft`、`published`、`archived` 基础上增加审核态，形成创建、编辑、提交审核、审核通过、驳回、发布、下线、归档闭环。
3. 增加内容质量检查，覆盖标题、摘要、标签、云厂商、行业、内容块、视频、CTA、权限配置等检查项。
4. 在资产列表和编辑页展示质量分、缺失项和可发布状态。

交付目标：管理员可以清楚判断哪些资产能发布、哪些需要处理、为什么不能发布。

### M2：前台体验增强

目标：让前台从“能浏览”变成“能决策”。

交付内容：

1. 强化资产库，补专题集合、热门资产、最新资产、推荐资产、筛选排序、空状态和结果解释。
2. 强化资产详情页，补适合谁用、如何使用、申请访问、收藏、分享、反馈、相关资产等决策辅助区。
3. 增加用户轻交互，包括收藏、最近浏览、问题反馈、访问申请。
4. 无交付权限时，不只显示提示，而是引导用户提交访问申请。

交付目标：用户可以快速判断资产是否适合自己，并知道下一步如何使用或申请访问。

### M3：可信度固化

目标：让平台从“有功能”变成“有证据”。

交付内容：

1. 增加基础数据看板，聚合浏览、收藏、申请、反馈、发布、审核、质量分趋势等指标。
2. 增加操作审计，记录资产变更、审核发布、权限策略、用户管理等关键动作。
3. 增加运营复盘视图，展示低质量资产、热门资产、待处理申请、异常权限命中和审核时长。
4. 为后续智能化能力沉淀事件和质量数据基础。

交付目标：平台能用数据证明内容可靠、流程可控、运营可复盘。

## 4. 功能模块设计

### 4.1 运营工作台

运营工作台替代当前偏静态的后台总览，成为管理员进入后台后的第一处理入口。

页面能力：

1. 平台总览：资产总数、已发布资产、待审核资产、低质量资产、待处理访问申请。
2. 待办队列：待审核、被驳回待修改、质量不达标、待处理申请。
3. 内容健康榜：按质量分、缺失项、更新时间展示需要运营关注的资产。
4. 近期动态：近期发布、审核、归档、访问申请处理记录。
5. 快捷入口：新建资产、进入资产列表、管理模板、查看申请队列。

建议 API：

1. `GET /api/v1/admin/operations/overview`
2. `GET /api/v1/admin/operations/tasks`
3. `GET /api/v1/admin/operations/recent-activities`

数据策略：优先做聚合查询，不先建设复杂 BI 表。

### 4.2 资产生命周期与审核流

在现有资产发布状态基础上补审核状态和审核记录，形成真实运营流程。

状态建议：

1. `draft`：草稿，可编辑，不出现在前台。
2. `reviewing`：已提交审核，编辑受限或需撤回后编辑。
3. `rejected`：审核驳回，需要编辑者修改后重新提交。
4. `published`：已发布，出现在前台公开资产库。
5. `archived`：已归档，不出现在前台，但后台保留。

关键动作：

1. 提交审核：`POST /api/v1/admin/assets/{id}/submit-review`
2. 审核通过：`POST /api/v1/admin/assets/{id}/approve`
3. 审核驳回：`POST /api/v1/admin/assets/{id}/reject`
4. 发布：复用或增强 `POST /api/v1/admin/assets/{id}/publish`
5. 下线：复用或增强 `POST /api/v1/admin/assets/{id}/unpublish`
6. 归档：复用 `POST /api/v1/admin/assets/{id}/archive`

新增数据：

1. `asset_review_records`：记录资产、审核人、动作、原因、时间。
2. 资产表增加或规范审核状态字段，避免状态含义混乱。

### 4.3 内容质量治理

内容质量治理用于把“能不能发布”从人工主观判断变成系统可解释规则。

检查维度：

1. 基础信息：标题、slug、摘要、副标题。
2. 分类信息：云厂商、行业、技术标签、资产类型。
3. 内容结构：至少一个可见内容块，内容块字段合法。
4. 媒体完整度：视频 URL、封面、主视频配置。
5. 转化信息：CTA、使用说明、适用角色、价值摘要。
6. 权限配置：交付内容是否设置可访问角色或用户。

建议 API：

1. `GET /api/v1/admin/assets/{id}/quality-check`
2. `POST /api/v1/admin/assets/{id}/quality-check/recalculate`

数据策略：

1. 早期可运行时计算质量分，降低迁移成本。
2. M3 再考虑持久化到 `asset_quality_scores`，用于趋势看板。

### 4.4 前台发现体验

前台发现体验基于现有资产库筛选和分页能力继续增强，让资产库更像真实内容平台。

页面能力：

1. 专题集合：例如 AI Transformation、Cloud Migration、Industry Solutions。
2. 推荐区：根据云厂商、行业、技术标签或编辑精选推荐。
3. 热门和最新：基于浏览、收藏、发布时间排序。
4. 结果解释：显示当前筛选命中原因和空状态建议。
5. 入口分发：首页、资产库、详情页均能进入专题或推荐内容。

建议 API：

1. `GET /api/v1/assets/collections`
2. `GET /api/v1/assets/collections/{slug}`
3. `GET /api/v1/assets/recommended`
4. 扩展 `GET /api/v1/assets` 的排序参数。

新增数据：

1. `asset_collections`
2. `asset_collection_items`

### 4.5 资产详情转化

详情页从内容展示页升级为用户决策页。

页面能力：

1. 适合谁用：展示目标角色、行业、场景、云厂商。
2. 如何使用：展示使用步骤、前置条件、交付入口。
3. 转化动作：收藏、分享、申请访问、反馈问题。
4. 相关资产：同云厂商、同技术标签、同场景的相关资产。
5. 权限提示：无权限时解释为什么受限，并提供申请入口。

建议 API：

1. `POST /api/v1/assets/{id}/favorite`
2. `DELETE /api/v1/assets/{id}/favorite`
3. `POST /api/v1/assets/{id}/feedback`
4. `GET /api/v1/assets/{id}/related`

新增数据：

1. `asset_favorites`
2. `asset_feedback`
3. `analytics_events` 中记录详情浏览、CTA 点击、分享等事件。

### 4.6 访问申请与反馈

访问申请用于打通前台受限内容与后台运营处理。

前台能力：

1. 无权限访问交付内容时展示申请表。
2. 申请表包含用途、所属角色、预期使用方式。
3. 重复申请时展示当前处理状态。

后台能力：

1. 访问申请列表。
2. 申请详情。
3. 通过、拒绝、备注。
4. 处理后影响用户可访问的资产交付内容。

建议 API：

1. `POST /api/v1/access-requests`
2. `GET /api/v1/me/access-requests`
3. `GET /api/v1/admin/access-requests`
4. `POST /api/v1/admin/access-requests/{id}/approve`
5. `POST /api/v1/admin/access-requests/{id}/reject`

新增数据：

1. `access_requests`
2. 字段包括申请人、资产、用途、状态、处理人、处理原因、处理时间。

### 4.7 数据看板与审计

数据看板和审计用于支撑平台可信度。

看板指标：

1. 内容指标：资产总数、发布资产、待审核资产、低质量资产。
2. 体验指标：浏览、收藏、分享、反馈、访问申请。
3. 流程指标：平均审核时长、驳回次数、发布频率、申请处理时长。
4. 质量指标：平均质量分、缺失项分布、低质量资产趋势。
5. 治理指标：权限命中、拒绝访问、策略变更、管理员操作。

审计事件：

1. 资产创建、编辑、提交审核、审核通过、驳回、发布、下线、归档。
2. 访问申请提交、通过、拒绝。
3. 用户、角色、策略变更。
4. 登录和关键鉴权失败。

建议 API：

1. `GET /api/v1/admin/analytics/overview`
2. `GET /api/v1/admin/analytics/assets/{id}`
3. `GET /api/v1/admin/audit-logs`

新增数据：

1. `analytics_events`
2. `audit_logs`

## 5. 端到端数据流

1. 管理员创建资产，使用模板、内容块、视频和标签补齐基础内容。
2. 系统运行质量规则，给出缺失项、质量分和可发布状态。
3. 编辑者提交审核，资产进入运营工作台待审核队列，并生成审核记录。
4. 审核人通过或驳回；驳回必须填写原因，资产回到可编辑状态。
5. 通过审核后发布到前台资产库，并进入最近发布、专题、推荐候选池。
6. 用户通过搜索、筛选、专题、推荐进入详情页，系统记录浏览事件。
7. 用户查看价值、场景、视频、使用方式和相关资产，执行收藏、分享、反馈或申请访问。
8. 访问申请进入后台队列，管理员审批后影响用户可见的交付内容。
9. 看板聚合浏览、收藏、申请、反馈、质量、审核时长和发布表现，形成运营改进依据。

## 6. 权限边界

1. 匿名用户：可浏览公开已发布资产和销售向内容。
2. 登录用户：可收藏、反馈、申请访问交付内容。
3. 编辑者：可创建和编辑草稿、提交审核。
4. 审核或运营角色：可审核、发布、下线、处理访问申请。
5. 管理员：可管理用户、角色、策略、审计与系统配置。

权限策略应继续复用现有 RBAC 和策略模拟能力。新增流程动作需要映射为明确权限点，例如 `asset:submit_review`、`asset:approve`、`asset:reject`、`access_request:review`、`audit_log:read`。

## 7. 错误处理

1. 审核或发布失败返回结构化原因，例如缺标题、缺标签、无可见内容块、权限配置不完整。
2. 访问申请失败区分未登录、无权限、重复申请、资产不存在、管理员已处理。
3. 前台资产加载失败保留搜索和筛选状态，并提供重试和返回资产库入口。
4. 审核驳回必须填写原因，避免流程记录不可解释。
5. 看板接口失败时显示局部错误，不影响后台其他页面可用性。

## 8. 测试策略

后端测试：

1. 资产状态流：草稿、提交审核、通过、驳回、发布、下线、归档。
2. 质量检查：缺失字段、可见内容块、视频、权限配置。
3. 访问申请：创建、重复申请、审批通过、审批拒绝。
4. 审计事件：关键后台动作会写入审计日志。
5. 聚合指标：运营概览和看板统计返回稳定结构。

前端测试：

1. 资产库筛选、排序、专题集合和空状态。
2. 详情页收藏、反馈、申请访问、相关资产。
3. 后台审核队列、质量面板、访问申请处理。
4. 权限边界下的按钮显隐和错误提示。
5. 看板局部加载、错误和空状态。

端到端场景：

1. 创建资产 → 质量检查 → 提交审核 → 审核通过 → 发布到前台。
2. 用户进入详情页 → 收藏 → 申请访问 → 管理员审批 → 用户获得交付内容访问。
3. 后台变更策略或发布资产 → 审计日志记录 → 看板指标更新。

## 9. 落地边界

本季度先做“平台可信度”，不做以下高复杂度能力：

1. 复杂工作流引擎。
2. 企业级审批流和多级会签。
3. 全量埋点平台。
4. 智能推荐中台。
5. 个性化首页。
6. 高级 BI 报表搭建器。

这些能力可在本季度形成数据基础后，作为后续阶段扩展。

## 10. 建议实施顺序

1. 建立审核状态与审核记录模型。
2. 增加质量检查服务和发布前校验。
3. 升级后台运营工作台和资产列表状态操作。
4. 增强前台资产库和详情页转化动作。
5. 建立访问申请模型和后台处理队列。
6. 增加 analytics events 和 audit logs。
7. 汇总运营看板和资产表现页。

## 11. 验收标准

1. 管理员能从工作台看到待处理事项、质量风险和近期运营动态。
2. 资产可以完成创建、检查、提交审核、审核、发布、下线、归档闭环。
3. 发布失败或审核失败能解释原因。
4. 用户能在详情页完成收藏、反馈和访问申请。
5. 管理员能处理访问申请，并留下处理记录。
6. 数据看板能展示内容、体验、流程和治理基础指标。
7. 审计日志能记录关键后台动作。
8. 新增核心流程具备后端、前端或端到端测试覆盖。
